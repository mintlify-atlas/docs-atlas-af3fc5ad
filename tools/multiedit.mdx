The MultiEdit tool efficiently enters text into multiple input fields at once. It's ideal for form filling, batch data entry, and multi-field updates.

## Overview

MultiEdit clicks each field, clears existing content, and types the new text. All fields are processed in sequence with automatic focus management.

<Note>
  Each field is cleared before typing. To preserve existing content, use individual Type tool calls with `clear=False`.
</Note>

## Parameters

<ParamField path="locs" type="array">
  List of `[x, y, text]` tuples for coordinate-based input.
  
  **Example:** `[[100, 200, "John"], [100, 250, "Doe"], [100, 300, "john@email.com"]]`
</ParamField>

<ParamField path="labels" type="array">
  List of `[label, text]` tuples for label-based input.
  
  **Example:** `[[5, "John"], [7, "Doe"], [9, "john@email.com"]]`
</ParamField>

<Note>
  You must provide **either** `locs` or `labels`, but not both.
</Note>

## Return Values

Returns a confirmation string listing all edited fields:
```
Multi-edited elements at: (100,200) with text 'John', (100,250) with text 'Doe', (100,300) with text 'john@email.com'
```

## Examples

### Form Filling with Labels

```python
# 1. Take snapshot to identify form fields
Snapshot(use_annotation=True)

# 2. Fill all fields at once
MultiEdit(
    labels=[
        [5, "John"],
        [7, "Doe"],
        [9, "john.doe@email.com"],
        [11, "555-1234"],
        [13, "123 Main St"]
    ]
)
```

### Form Filling with Coordinates

```python
# Fill registration form using exact coordinates
MultiEdit(
    locs=[
        [300, 200, "Username123"],
        [300, 250, "SecurePass456!"],
        [300, 300, "SecurePass456!"],  # Confirm password
        [300, 350, "user@example.com"]
    ]
)
```

## Field Processing Order

Fields are processed **sequentially** in the order provided:
1. Click field
2. Clear existing text (`Ctrl+A` → `Backspace`)
3. Type new text
4. Move to next field

<Warning>
  If a field triggers validation or auto-save on blur, it will execute before the next field is focused.
</Warning>

## Text Handling

- **Special characters:** Automatically escaped (e.g., `{`, `}`, newlines)
- **Line breaks:** `\n` converted to `{Enter}`
- **Tabs:** `\t` converted to `{Tab}`
- **Clearing:** Always uses `Ctrl+A` → `Backspace` sequence

## Error Handling

```python
# Missing parameters
"Either locs or labels must be provided."

# Invalid label format
"Each label item must be [label, text]. Invalid: [5]"

# Label not found
"Failed to process label item [99, 'text']: IndexError"

# No snapshot
"Desktop state is empty. Please call Snapshot first."
```

## Best Practices

<Tip>
  **Call Snapshot immediately before** MultiEdit to ensure fresh UI element data.
</Tip>

<Tip>
  **Group related fields** together in a single MultiEdit call for speed and atomicity.
</Tip>

<Tip>
  **Verify submission** with a Snapshot after form fill to ensure no validation errors.
</Tip>

<Warning>
  **Watch for auto-complete:** Some fields trigger dropdowns or suggestions. If interference occurs, use individual Type calls with `press_enter=False`.
</Warning>

## Labels vs Coordinates

| Approach | Best For | Trade-offs |
|----------|----------|------------|
| **Labels** | Dynamic forms | Requires Snapshot, adapts to layout changes |
| **Coordinates** | Fixed layouts | Faster, no Snapshot needed, brittle to UI changes |

## When to Use

**Use MultiEdit when:**
- Filling 3+ fields in a form
- All fields accept text input
- Fields don't have complex validation
- Speed is important

**Use individual Type when:**
- Field has special behavior (e.g., date picker, auto-complete)
- Need to preserve existing content
- Validation must be checked between fields
- Need `press_enter=True` for submission

## Performance

Typical timing per field:
- Click: 0.1s
- Clear: 0.5s
- Type: 0.02s per character
- **Total per field:** ~1-2 seconds

<Tip>
  For 5 fields, MultiEdit takes ~5-10 seconds vs ~15-25 seconds with individual Type calls.
</Tip>

## Limitations

- All fields are cleared before typing (no append mode)
- No field-specific options (e.g., `press_enter`, `caret_position`)
- Sequential processing only (no parallelization)
- Requires all fields to be visible on screen
