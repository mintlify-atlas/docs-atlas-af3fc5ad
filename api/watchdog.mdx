## Overview

The WatchDog service (`watchdog/service.py`) monitors Windows UI Automation events in a dedicated background thread. It tracks focus changes, structure changes, and property changes to keep the UI state current without polling.

## Architecture

WatchDog runs in a Single-Threaded Apartment (STA) thread pumping COM events. It supports three event types:

- **Focus events**: Track which UI element has keyboard focus
- **Structure events**: Detect when UI elements are added/removed
- **Property events**: Monitor when element properties change (name, value, toggle state, etc.)

## Core Methods

### start()

Start the background event monitoring thread.

```python
def start(self)
```

Starts a daemon thread named `"WatchDogThread"` that:
1. Initializes COM with `CoInitialize()`
2. Registers configured event handlers
3. Pumps events with `PumpEvents(0.1)`
4. Cleans up handlers on exit

### stop()

Stop the background thread and unregister handlers.

```python
def stop(self)
```

Waits up to 2 seconds for thread to terminate gracefully.

### set_focus_callback()

<ParamField path="callback" type="Callable[[Any, StructureChangeType, list], None] | None">

<ParamField path="element" type="IUIAutomationElement | None">

**Structure change types:**
- `ChildAdded`: New element added
- `ChildRemoved`: Element removed
- `ChildrenInvalidated`: Children list changed
- `ChildrenBulkAdded`: Multiple children added
- `ChildrenBulkRemoved`: Multiple children removed

## Context Manager Support

WatchDog supports context manager protocol:

```python

    watchdog.set_focus_callback(on_focus)
    # ... do work ...
```

## Usage Examples

### Basic Focus Monitoring

```python
from windows_mcp.watchdog.service import WatchDog
from windows_mcp.uia import Control
import time

def on_focus(sender):
    element = Control.CreateControlFromElement(sender)
    print(f"Focus changed to: {element.Name}")

watchdog = WatchDog()
watchdog.set_focus_callback(on_focus)
watchdog.start()

try:
    time.sleep(60)  # Monitor for 60 seconds
finally:
    watchdog.stop()
```

### Monitor Specific Window

```python
from windows_mcp.watchdog.service import WatchDog
import windows_mcp.uia as uia

def on_structure_change(sender, change_type, runtime_id):
    print(f"Structure change: {change_type}")

# Get specific window to monitor
window = uia.WindowControl(Name="Calculator")
if window.Exists():
    watchdog = WatchDog()
    watchdog.set_structure_callback(
        callback=on_structure_change,
        element=window.Element  # Monitor only Calculator
    )
    watchdog.start()
```

### Monitor Property Changes

```python
from windows_mcp.watchdog.service import WatchDog
from windows_mcp.uia import PropertyId

def on_property_change(sender, property_id, new_value):
    if property_id == PropertyId.NameProperty:
        print(f"Name changed to: {new_value}")
    elif property_id == PropertyId.ValueProperty:
        print(f"Value changed to: {new_value}")

watchdog = WatchDog()
watchdog.set_property_callback(
    callback=on_property_change,
    property_ids=[PropertyId.NameProperty, PropertyId.ValueProperty]
)
watchdog.start()
```

## Performance Notes

- **STA Threading**: Runs in Single-Threaded Apartment to avoid COM marshaling overhead (watchdog/service.py:90)
- **Event Pumping**: Uses `PumpEvents(0.1)` for 100ms polling interval (watchdog/service.py:196)
- **Debouncing**: The Tree service debounces duplicate focus events within 1 second (tree/service.py:615)
- **Handler Registration**: Handlers are dynamically registered/unregistered as callbacks are set/cleared (watchdog/service.py:94-193)

## Troubleshooting

### Events Not Firing

1. Ensure `start()` was called
2. Check callback is not `None`
3. Verify element scope (Desktop vs specific window)
4. Some applications don't fire UI Automation events

### Thread Not Stopping

If `stop()` times out after 2 seconds:
- Thread may be blocked in event handler
- Ensure handlers don't block indefinitely
- Use daemon threads for background work in handlers

### COM Errors

- `CoInitialize()` called automatically in thread
- Don't call UI Automation from handler callback (marshaling issues)
- Queue events and process in main thread if needed
